## 코드 품질 관리

실수 사례를 보기 전에, 코드 품질을 지키기 위한 기법들을 먼저 훑는 챕터다. 코드 리뷰, 코드 스타일, 정적 분석 등이 어떻게 버그를 줄이는지 정리한다.

### 코드 리뷰와 페어 프로그래밍

페어 프로그래밍은 익스트림 프로그래밍(XP) 방법론에서 등장한 기법이다. 두 명이 함께 작업하는 방식으로, 하나의 워크스테이션을 공유하거나 나란히 같이 작업한다. 한 명이 코드를 작성하고 다른 한 명이 그 자리에서 검토하므로, 코드 작성과 리뷰를 동시에 거치는 셈이다.

### 코드 스타일

자바는 하나의 목표를 여러 방식으로 작성할 수 있다. 괄호를 생략하거나, 로컬 변수를 전부 `final`로 지정하거나, 들여쓰기를 마음대로 할 수 있다. 그래서 어떤 스타일로 통일하느냐가 가독성과 실수 방지에 영향을 준다.

대표적인 예가 `long` 리터럴이다. `10l`과 `10L` 둘 다 가능하지만, 소문자 `l`은 숫자 `1`과 혼동되기 쉽다. 그래서 보통 대문자 `L`을 써서 `10L`로 작성한다.

`if`문도 비슷한 이유로 난감할 때가 있다. 중괄호는 문법상 생략이 가능하지만, 생략하면 실수의 소지가 커진다.

```java
if (a > b)
    System.out.println("a is bigger");
```

이렇게 한 줄짜리는 동작은 한다. 문제는 여기서 줄을 하나 더 붙일 때다.

```java
if (a > b)
    System.out.println("a is bigger");
    System.out.println("this always runs"); // if와 무관하게 항상 실행됨
```

이건 **컴파일 에러가 아니다.** 들여쓰기 때문에 두 번째 출력도 `if`에 묶인 것처럼 보이지만, 실제로는 `if`의 본문은 첫 번째 문장 하나뿐이다. 두 번째 문장은 조건과 상관없이 항상 실행된다. 들여쓰기만 보고 착각하기 딱 좋은 논리 버그다.

```java
if (a > b) {
    System.out.println("a is bigger");
    System.out.println("this runs only when a > b");
}
```

그래서 보통은 이렇게 중괄호를 써서 본문 범위를 명확히 한다. 한 줄이라도 중괄호를 붙이는 스타일을 강제하면 이런 실수를 원천 차단할 수 있다.

### 정적 분석

정적 분석은 코드를 실행하지 않고 소스만 분석해 버그 가능성이 높은 코드를 지목하는 기법이다. 일종의 자동화된 코드 리뷰라고 볼 수 있다. 사람이 리뷰하듯 의심스러운 패턴을 짚어주되, 사람을 완전히 대체하지는 못한다. 사람이라면 쉽게 알아챌 오류를 정적 분석기는 끝내 못 잡는 경우도 있기 때문이다.

분석 도구로는 IntelliJ IDEA(내장 인스펙션), SonarQube 등이 있고, 이 외에도 종류가 많다.

### 그 외

이 챕터에는 동적 분석, 뮤테이션 커버리지, 코드 어설션 등에 대한 설명도 이어진다. 코드 품질 파트는 개론 성격이라 여기까지 읽고 넘어간다.
